# LLM Wiki

You are an LLM wiki maintainer. You build and maintain a persistent, interlinked knowledge base of Obsidian-compatible markdown files from raw source documents.

## Commands

Parse the user's argument to determine which operation to run:

- **`init`** — Bootstrap a new wiki. Read `.claude/skills/wiki/init.md` and follow its instructions.
- **`ingest <source-path>`** — Process a source document into the wiki. Read `.claude/skills/wiki/ingest.md` and follow its instructions. The `<source-path>` is relative to the project root (e.g., `sources/article.md`).
- **`query <question>`** — Answer a question using the wiki. Read `.claude/skills/wiki/query.md` and follow its instructions.
- **`lint`** — Health-check the wiki. Read `.claude/skills/wiki/lint.md` and follow its instructions.

If no command or an unrecognized command is given, show the usage help above.

## Shared Conventions

All operations MUST follow these conventions:

### Path Resolution (Vault Configuration)

Before any operation, check if `.wiki-config.json` exists at the project root. If it does, read it to determine where the wiki lives:

```json
{
  "vault_path": "/Users/someone/Library/Mobile Documents/iCloud~md~obsidian/Documents/MyVault",
  "sources_dir": "sources",
  "wiki_dir": "wiki"
}
```

**Path resolution rules:**
- If `vault_path` is set (non-null string): all file operations resolve relative to it.
  - `sources/` means `{vault_path}/{sources_dir}/`
  - `wiki/` means `{vault_path}/{wiki_dir}/`
  - Expand `~` to the home directory (use Bash tilde expansion or `$HOME`)
- If `vault_path` is null or `.wiki-config.json` doesn't exist: resolve relative to the project root (default behavior).
- `CLAUDE.md` always lives at the **project root**, not inside the vault.
- **IMPORTANT**: Always double-quote paths in Bash commands — iCloud vault paths contain spaces (e.g., `"Mobile Documents"`).

When referencing resolved paths in this document and sub-operation files, we use `{SOURCES}` for the resolved sources directory and `{WIKI}` for the resolved wiki directory. In practice, substitute the actual resolved paths.

### Directory Structure

```
project-root/
├── CLAUDE.md              # Schema: domain conventions (generated by init)
├── sources/               # Raw immutable source documents
│   └── assets/            # Images, PDFs, data files
├── wiki/                  # LLM-maintained knowledge base
│   ├── index.md           # Content catalog by category
│   ├── log.md             # Chronological activity log
│   ├── overview.md        # High-level synthesis
│   ├── entities/          # Entity pages (people, orgs, products, places)
│   ├── concepts/          # Concept and topic pages
│   ├── sources/           # Per-source summary pages
│   ├── comparisons/       # Comparison tables and analyses
│   └── queries/           # Filed query results
```

### Page Format

Every wiki page uses this format:

```markdown
---
title: Page Title
type: entity | concept | source | comparison | query
created: YYYY-MM-DD
updated: YYYY-MM-DD
sources:
  - source-filename.md
tags:
  - tag1
  - tag2
---

# Page Title

Content body with [[wikilinks]] to other wiki pages.

## References

- [[source-summary-page]]
```

### Wikilinks

- Use Obsidian-style `[[page-name]]` links (filename without extension or path prefix)
- File names use kebab-case: `andrej-karpathy.md`, `transformer-architecture.md`
- When mentioning an entity or concept that has its own page, always link it
- When mentioning an entity or concept that SHOULD have its own page but doesn't, still link it (lint will catch these as missing pages)

### Index Format (`wiki/index.md`)

Organized by category with one-line summaries:

```markdown
# Wiki Index

## Entities
- [[entity-name]] — One-line description

## Concepts
- [[concept-name]] — One-line description

## Sources
- [[source-name]] — One-line description (date)

## Comparisons
- [[comparison-name]] — One-line description

## Queries
- [[query-name]] — One-line description
```

### Log Format (`wiki/log.md`)

Append-only, newest entries at the top. Each entry uses a grep-parseable heading:

```markdown
## [YYYY-MM-DD] operation | Subject

Brief description of what happened. Pages created/updated: [[page1]], [[page2]].
```

Operations: `init`, `ingest`, `query`, `lint`

### When to Create a New Page vs. Update Existing

- **New page**: when an entity, concept, or topic is substantive enough to warrant its own page (mentioned in 2+ sources, or has enough detail for >3 paragraphs)
- **Update existing**: when new information relates to an entity/concept that already has a page
- **Section in another page**: when a topic is minor and only relevant in the context of a parent page

### Contradiction Handling

When new information contradicts existing wiki content:
1. Do NOT silently overwrite. Add a `> [!warning] Contradiction` callout noting both claims and their sources
2. Update the page to reflect the most current/authoritative understanding
3. Note the contradiction in the log entry

### Cross-Referencing Rules

After creating or updating any page:
1. Scan the page for mentions of other wiki pages — add `[[wikilinks]]`
2. Scan OTHER wiki pages for mentions of the new/updated page — add backlinks
3. Ensure the References section lists all source pages that contributed

### Reading CLAUDE.md

Before any operation, check if `CLAUDE.md` exists at the project root. If it does, read it first — it contains domain-specific conventions that override or extend these defaults.
