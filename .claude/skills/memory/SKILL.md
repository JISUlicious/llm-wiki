# Agent Memory

You are an agent memory manager. You build and maintain a persistent, interlinked memory store of Obsidian-compatible markdown files from raw inputs — conversations, logs, observations, and notes.

## Commands

Parse the user's argument to determine which operation to run:

- **`init`** — Bootstrap a new memory store. Read `.claude/skills/memory/init.md` and follow its instructions.
- **`record <source-path>`** — Record an experience into memory. Read `.claude/skills/memory/record.md` and follow its instructions. The `<source-path>` is relative to the project root (e.g., `inputs/conversation.md`).
- **`query <question>`** — Recall information from memory. Read `.claude/skills/memory/query.md` and follow its instructions.
- **`reflect [topic | --global]`** — Generate meta-observations from accumulated memory. Read `.claude/skills/memory/reflect.md` and follow its instructions.
- **`consolidate [topic]`** — Merge related episodes into reusable schemas. Read `.claude/skills/memory/consolidate.md` and follow its instructions.
- **`lint`** — Health-check the memory store. Read `.claude/skills/memory/lint.md` and follow its instructions.

If no command or an unrecognized command is given, show the usage help above.

## Shared Conventions

All operations MUST follow these conventions:

### Path Resolution (Vault Configuration)

Before any operation, check if `.memory-config.json` exists at the project root. If it does, read it to determine where the memory store lives:

```json
{
  "vault_path": "/Users/someone/Library/Mobile Documents/iCloud~md~obsidian/Documents/MyVault",
  "inputs_dir": "inputs",
  "memory_dir": "memory",
  "mode": "manual",
  "auto_consolidate_threshold": 5,
  "stale_days": 30,
  "access_count_prune_threshold": 2
}
```

**Path resolution rules:**
- If `vault_path` is set (non-null string): all file operations resolve relative to it.
  - `inputs/` means `{vault_path}/{inputs_dir}/`
  - `memory/` means `{vault_path}/{memory_dir}/`
  - Expand `~` to the home directory (use Bash tilde expansion or `$HOME`)
- If `vault_path` is null or `.memory-config.json` doesn't exist: resolve relative to the project root (default behavior).
- `CLAUDE.md` always lives at the **project root**, not inside the vault.
- **IMPORTANT**: Always double-quote paths in Bash commands — iCloud vault paths contain spaces (e.g., `"Mobile Documents"`).

**Config fields:**
- `mode`: `"manual"` (human reviews all operations, default) or `"auto"` (agent self-manages — skips confirmation prompts, proceeds without human-in-the-loop)
- `auto_consolidate_threshold`: when N episodes share a tag cluster, suggest consolidation (default 5)
- `stale_days`: memories not accessed in this many days are candidates for archival (default 30)
- `access_count_prune_threshold`: memories accessed fewer than this many times AND stale are pruning candidates (default 2)

When referencing resolved paths in this document and sub-operation files, we use `{INPUTS}` for the resolved inputs directory and `{MEMORY}` for the resolved memory directory. In practice, substitute the actual resolved paths.

### Directory Structure

```
project-root/
├── CLAUDE.md              # Schema: memory domain conventions (generated by init)
├── inputs/                # Raw immutable input documents
│   ├── conversations/     # Chat logs, transcripts
│   ├── logs/              # System logs, error traces
│   ├── observations/      # Manual notes, environment snapshots
│   └── assets/            # Images, screenshots, data files
├── memory/                # Agent-maintained memory store
│   ├── index.md           # Content catalog by category
│   ├── log.md             # Chronological activity log
│   ├── overview.md        # High-level synthesis of what the agent knows
│   ├── episodes/          # Per-input episode summaries (episodic memory)
│   ├── actors/            # Agents, tools, users, systems, APIs (semantic memory)
│   ├── schemas/           # Strategies, patterns, procedures, heuristics (semantic/procedural memory)
│   ├── evaluations/       # Comparative analyses, what-worked-vs-what-didn't
│   └── reflections/       # Filed meta-observations and synthesized lessons
```

### Page Format

Every memory page uses this format:

```markdown
---
title: Page Title
type: episode | actor | schema | evaluation | reflection
created: YYYY-MM-DD
updated: YYYY-MM-DD
sources:
  - input-filename.md
tags:
  - tag1
  - tag2
status: draft | active | consolidated | archived
importance: high | medium | low
confidence: high | medium | low | uncertain
last_accessed: YYYY-MM-DD
access_count: 0
memory_type: episodic | semantic | procedural
validity_window: permanent | YYYY-MM-DD
supersedes: []
---

# Page Title

Content body with [[wikilinks]] to other memory pages.

## References

- [[episode-summary-page]]
```

**Frontmatter field guide:**
- `status`: lifecycle state — `draft` (incomplete), `active` (current, in use), `consolidated` (merged into a broader schema), `archived` (expired or superseded)
- `confidence`: how reliable — `high` (corroborated by multiple episodes), `medium` (2-3 episodes), `low` (single episode), `uncertain` (contradicted or unverified)
- `last_accessed`: updated every time the page is read during query or reflect
- `access_count`: incremented every time the page is read during query or reflect
- `memory_type`: `episodic` (specific events), `semantic` (factual knowledge), `procedural` (how-to knowledge)
- `validity_window`: `permanent` for timeless facts, or a `YYYY-MM-DD` expiration date for time-bound info
- `supersedes`: list of page filenames this page replaces (audit trail for consolidation)

### Wikilinks

- Use Obsidian-style `[[page-name]]` links (filename without extension or path prefix)
- File names use kebab-case: `api-timeout-debugging.md`, `claude-code-agent.md`
- When mentioning an actor or schema that has its own page, always link it
- When mentioning an actor or schema that SHOULD have its own page but doesn't, still link it (lint will catch these as missing pages)

### Index Format (`memory/index.md`)

Organized by category with one-line summaries:

```markdown
# Memory Index

## Actors
- [[actor-name]] — One-line description

## Schemas
- [[schema-name]] — One-line description

## Episodes
- [[episode-name]] — One-line description (date)

## Evaluations
- [[evaluation-name]] — One-line description

## Reflections
- [[reflection-name]] — One-line description
```

### Log Format (`memory/log.md`)

Append-only, newest entries at the top. Each entry uses a grep-parseable heading:

```markdown
## [YYYY-MM-DD] operation | Subject

Brief description of what happened. Pages created/updated: [[page1]], [[page2]].
```

Operations: `init`, `record`, `query`, `reflect`, `consolidate`, `lint`

### When to Create a New Page vs. Update Existing

- **New page**: when an actor, schema, or topic is substantive enough to warrant its own page (mentioned in 2+ episodes, or has enough detail for >3 paragraphs)
- **Update existing**: when new information relates to an actor/schema that already has a page
- **Section in another page**: when a topic is minor and only relevant in the context of a parent page

### Contradiction Handling

When new information contradicts existing memory content:
1. Do NOT silently overwrite. Add a `> [!warning] Contradiction` callout noting both claims and their sources
2. Lower the `confidence` of the affected page if warranted
3. Update the page to reflect the most current/authoritative understanding
4. Note the contradiction in the log entry

### Cross-Referencing Rules

After creating or updating any page:
1. Scan the page for mentions of other memory pages — add `[[wikilinks]]`
2. Scan OTHER memory pages for mentions of the new/updated page — add backlinks
3. Ensure the References section lists all episode pages that contributed

### Git Workflow

After any operation that modifies memory files (record, query with filing, consolidate, reflect, lint with auto-fix), offer to create a git commit:

> "Would you like me to commit these changes? Message: `record: {{Source Title}} — {{N}} pages created, {{N}} updated`"

If the user accepts:
1. Stage only modified files under `memory/` and `CLAUDE.md` (not `.claude/` skill files)
2. Commit with the offered message
3. Do not push unless the user explicitly asks

Commit message format mirrors the log entry:
- `record: Source Title — N pages created, N updated`
- `query: Question slug — filed as reflection`
- `consolidate: Topic — N episodes merged into schema`
- `reflect: Topic — reflection filed`
- `lint: N errors fixed, N warnings fixed`

This is opt-in — always ask, never auto-commit. In `auto` mode, skip the prompt and commit automatically.

### Scaling

At small scale (~100 inputs, hundreds of pages), the index file + Grep is sufficient for navigation. As the memory store grows beyond this:

- **[qmd](https://github.com/tobi/qmd)** — a local search engine for markdown files with hybrid BM25/vector search and LLM re-ranking, all on-device. It has both a CLI (so the LLM can shell out to it) and an MCP server (so the LLM can use it as a native tool). Set it up to index `{MEMORY}/` for full-text search across all memory pages.
- If qmd or another search tool is available, the query operation should prefer it over Grep for finding relevant pages in Step 1.
- The index file remains useful as a human-browsable catalog even when search is available.

Before any operation, check if `CLAUDE.md` exists at the project root. If it does, read it first — it contains domain-specific conventions that override or extend these defaults.
